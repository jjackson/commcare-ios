# App Store Preparation — Design Spec

**Date:** 2026-03-20
**Phase:** 4, Wave 5 (Issue #257)
**Goal:** Make the app TestFlight-ready under a personal developer account.

## Decisions

- **Bundle ID:** `org.marshellis.commcare.ios` (personal account, avoids conflict with Dimagi's eventual production ID)
- **Distribution:** TestFlight only — no App Store submission yet
- **Signing:** Automatic via Xcode, no CI signing for now

- **ConnectIdApi.APPLICATION_ID:** Keep as `org.commcare.ios` — this is a server-registered identifier for Dimagi's ConnectID service, not the Xcode bundle ID

## 2. App Icon

The existing `icon-1024.png` in Assets.xcassets already contains the correct CommCare interlocking-loops icon at 1024x1024. Verify it matches the official branding favicon — if it does, no change needed.

If replacement is needed, render from the SVG source (`docs/branding/CommCare.svg`) at 1024x1024 to avoid upscaling artifacts. Do NOT upscale the 432x432 PNG favicon.

Modern iOS 16+ only needs the single universal 1024x1024 — the OS auto-generates smaller sizes. The current `Contents.json` is already configured correctly for this.

## 3. Launch Screen

Use the Info.plist `UILaunchScreen` dictionary approach (modern, no storyboard needed for iOS 16+):

```xml
<key>UILaunchScreen</key>
<dict>
    <key>UIImageName</key>
    <string>LaunchLogo</string>
    <key>UIColorName</key>
    <string>LaunchBackground</string>
</dict>
```

Add to `Assets.xcassets`:
- `LaunchLogo` image set — CommCare Full Color Logo from `docs/branding/CommCare-Full-Color-Logo.png`
- `LaunchBackground` color set — white

This avoids adding a storyboard file (Interface Builder XML is hard to generate correctly). The logo image must be committed to the repo (not gitignored) since it's a build asset.

Remove the existing empty `UILaunchScreen` dict from Info.plist and replace with the above.

Update `project.yml` `sources` to include any new asset catalog entries.

## 4. Privacy Manifest

Create `app/iosApp/iosApp/PrivacyInfo.xcprivacy`.

Declare API usage reasons:
- `NSPrivacyAccessedAPICategoryUserDefaults` — reason: `CA92.1` ("Access info from same app, app group, or library")
- `NSPrivacyAccessedAPICategoryFileTimestamp` — reason: `C617.1` ("Access file timestamps inside the app container")

No tracking domains. No third-party SDKs with privacy requirements.

Add to `project.yml` `sources` so xcodegen includes it in the bundle.

Existing `NS*UsageDescription` keys in Info.plist are already complete — no changes needed.

## 5. Version & Build Number

- **Marketing version (CFBundleShortVersionString):** `1.0` — stays fixed for TestFlight
- **Build number (CFBundleVersion):** Auto-incremented via git commit count

Add a post-build script in `project.yml` under `targets.CommCare.postBuildScripts`:
```yaml
postBuildScripts:
  - name: Set Build Number from Git
    script: |
      COMMIT_COUNT=$(git rev-list --count HEAD)
      /usr/libexec/PlistBuddy -c "Set :CFBundleVersion $COMMIT_COUNT" "${TARGET_BUILD_DIR}/${INFOPLIST_PATH}"
```

This ensures every build has a unique, monotonically increasing build number tied to the commit history. TestFlight displays `1.0 (347)` etc.

## 6. TestFlight Setup Doc

Write `docs/plans/testflight-setup.md` covering:
- Prerequisites: Apple Developer account, Xcode with signing identity
- Archive: Product > Archive in Xcode, select "Distribute App" > App Store Connect > Upload

- Invite testers: App Store Connect > TestFlight > Internal/External Testing > Add testers

No CI automation for now — manual Xcode upload is appropriate for skunkworks stage.
