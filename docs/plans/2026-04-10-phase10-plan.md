# Phase 10: Quality Hardening + App Store

**Date:** 2026-04-10
**Predecessor:** Phase 9 (E2E UI Testing) — 15 bugs found and fixed, systematic AI-port test coverage gap identified
**Goal:** Close the remaining quality gaps surfaced by Phase 9, then submit to the App Store.

## Context

Phase 9's E2E testing revealed three systemic issues:

1. **ViewModel layer has no real tests.** The placeholder audit removed 63 fake tests but the remaining coverage is thin. Every ViewModel state field that's set from multiple code paths can silently regress (as demonstrated by #406 MSelect state, #416 CMP recomposition, #410 login race).

2. **iOS keychain `SecItemAdd` silently fails** in the Compose context. The K/N interop dual-path (`mapOf as CFDictionaryRef` vs `NSMutableDictionary + CFBridgingRetain`) doesn't reliably persist items. Worked around via SQLite DB fallback (#389), but the root cause is unknown and the workaround is less secure.

3. **Connect marketplace scaffolding gaps.** The messaging consent API was sending the wrong payload (#424). The `retrieve_messages` response wasn't being parsed correctly. Several Connect API scaffolds guessed at server response shapes.

## Streams

### Stream 1: ViewModel Test Backfill (3-5 days)

**Goal:** Every ViewModel mutable state field that's set from multiple code paths has a round-trip test.

**Method:** For each ViewModel in `app/src/commonMain/.../viewmodel/`:
1. List all `var ... by mutableStateOf` fields
2. Identify which are set from >1 code path (direct set + updateQuestions, direct set + loadThreads, etc.)
3. Write a test that exercises each code path and asserts the field survives

**Priority ViewModels (highest bug density in Phase 9):**
- `FormEntryViewModel` — draftTexts, questions, selectedChoices, constraintMessage, isComplete, isRepeatPrompt
- `LoginViewModel` — appState, currentApp, resolveDomain
- `MenuViewModel` — menuItems (badge count rendering)
- `DrawerViewModel` — hasConnectAccess, apps, seatedAppId
- `OpportunitiesViewModel` — opportunities, selectedOpportunity, downloadState
- `MessagingViewModel` — threads, hasConsented, availableChannels

**Acceptance:** Every ViewModel has at least one test file with real object construction (no `assertFalse(false)` placeholders). `./gradlew :app:jvmTest` passes.

### Stream 2: iOS Keychain Investigation (1-2 days)

**Goal:** Determine why `SecItemAdd` silently fails in the Compose onClick context and fix or document the root cause.

**Investigation plan:**
1. Add OSStatus return-value logging to `PlatformKeychainStore.store()` (already started)
2. Test keychain operations from a `LaunchedEffect` (outside the Compose onClick handler) to isolate whether the issue is timing, threading, or the K/N bridge
3. Test with `kSecAttrAccessibleAfterFirstUnlock` instead of `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`
4. Test with `SecItemUpdate` instead of delete+add
5. If the K/N bridge is the issue, consider using a Swift interop layer for keychain operations

**Acceptance:** Either the keychain works reliably (remove DB fallback) or the root cause is documented with a permanent workaround.

### Stream 3: App Store Submission (1-2 days, mostly manual)

**Goal:** Submit CommCare iOS to the App Store.

**Tasks (from `project_appstore_remaining.md`):**
1. Add App Store privacy labels (nutrition labels in App Store Connect)
2. Take iPad screenshot for App Store listing
3. Submit for review

**Prerequisite:** Streams 1 and 2 should be complete (or at least Stream 1) to ensure the app is stable.

### Stream 4: Connect API Hardening (1-2 days)

**Goal:** Fix the remaining Connect marketplace scaffolding gaps.

**Tasks:**
1. Verify `updateConsent` works when messaging channels ARE configured (needs server-side test data)
2. Wire up "Sign in to Personal ID" in nav drawer (currently `{ /* placeholder */ }`)
3. Verify learn app download flow when a learn app URL is configured
4. Test the delivery form submission path

**Dependency:** Requires Connect team to configure messaging channels and learn app URLs on the demo opportunities.

## Ordering

1. Stream 1 (ViewModel tests) — highest value, no external dependencies
2. Stream 2 (keychain) — can be done in parallel with Stream 1
3. Stream 4 (Connect) — blocked on server-side configuration
4. Stream 3 (App Store) — after Streams 1-2 complete

- [ ] iOS keychain root cause documented or fixed
