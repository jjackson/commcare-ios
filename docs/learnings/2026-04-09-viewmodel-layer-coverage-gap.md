# The ViewModel Layer Has No Tests

**Date:** 2026-04-09
**Context:** Phase 9 Wave 5a scouting, fifth bug in the same-session chain. `FormEntryViewModel.toggleMultiSelectChoice` was wiping its own state on every call because `updateQuestions()` didn't populate `selectedChoices` from the engine's accepted answer.

## The Bug

Tapping a checkbox on a select-multi question briefly highlighted the row but never rendered the box as checked. The engine accepted the answer (the `SelectMultiData` was submitted to `formSession.answerAtIndex` and returned `ANSWER_OK`), but the UI never showed it, so the user couldn't advance past the required question.

Root cause: `FormEntryViewModel.updateQuestions()` rebuilds the questions list from `formSession.getPrompts()` after every `answerQuestion()` call. The rebuilt `QuestionState` was missing `selectedChoices` — the field defaulted to `emptySet()`. Every tap went:

1. `toggleMultiSelectChoice(index, "fever")`
2. Copies `questions[index]` with `selectedChoices = {fever}` — transient in-memory state
3. Calls `answerQuestion(index, SelectMultiData([fever]))`
4. Engine returns `ANSWER_OK`
5. `answerQuestion` calls `updateQuestions()`
6. `updateQuestions()` rebuilds every `QuestionState` without `selectedChoices` → defaults to `emptySet()`
7. Compose recomposes → box renders unchecked
8. User is stuck

Fix: read `prompt.getAnswerValue()` in `updateQuestions()` and, when it's a `SelectMultiData`, extract the selected values. Engine is the source of truth — the UI should always reflect what the engine has accepted.

## The Gap This Exposes

**The ViewModel layer in `app/` has effectively no unit tests.** Not one of the existing ViewModel tests exercised `toggleMultiSelectChoice`, and the file that *looked* like it tested the ViewModel — `RepeatGroupTest.kt` — is a set of placeholder assertions that the author wrote with a comment explicitly acknowledging the problem:

```kotlin
@Test
fun testRepeatPromptStateInitiallyFalse() {
    // FormEntryViewModel.isRepeatPrompt starts as false
    // (cannot construct ViewModel without FormEntrySession, so test state defaults)
    val isRepeatPrompt = false
    assertFalse(isRepeatPrompt)
}
```

This is a test that asserts `false == false`. It doesn't construct a ViewModel. It doesn't call any ViewModel method. It is not a test. `FormNavigationTest.kt` in the same package follows the same pattern — asserting that `100f > 0` and that enum `entries.contains(SELECT_MULTI)` — rather than actually exercising the navigation logic.

The real coverage triangle looks like this:

|                    | Covered?                  |
|--------------------|---------------------------|
| Engine (commcare-core, XForm processing) | Yes — oracle tests, golden files, JVM test suite |
| Screen layer (Compose widgets, Maestro flows) | Yes — Maestro E2E on real simulators |
| **ViewModel layer (glue between engine and screens)** | **Essentially nothing.** |

When I needed to test `toggleMultiSelectChoice`, I had to write the scaffolding from scratch (load XML → build FormDef → build FormEntrySession → construct ViewModel → navigate → assert). That's a pattern that should have existed from Phase 2. Instead, the AI port produced assertion-free stubs and the lack of coverage was invisible until production E2E.

## Why the AI Port Produced Fake Tests

The existing tests look like real tests from a distance: they have `@Test` annotations, `assertEquals` calls, descriptive names. But they don't *do* anything. They were probably generated by an AI conversion process that saw "we need tests for this ViewModel" and produced code that technically compiles, technically has assertions, and technically "passes." The signal from `./gradlew :app:jvmTest` stayed green because there was nothing to fail.

This is the most expensive form of technical debt: **tests that look like tests but aren't.** They give the reviewer (human or AI) confidence that the code is covered, but they provide zero regression protection. The proper move would have been to mark them `@Ignore("TODO: needs test fixture")` or delete them entirely. Instead they sit in the repo, green, hiding the gap.

## Tests Added

`app/src/jvmTest/kotlin/org/commcare/app/viewmodel/MultiSelectStateTest.kt` — 4 tests, all use the real scaffolding:

1. **`selectedChoicesPersistAfterSingleToggle`** — minimal repro (one tap, one assertion).
2. **`selectedChoicesAccumulateAcrossMultipleToggles`** — three consecutive toggles accumulate.
3. **`togglingSelectedChoiceRemovesItFromState`** — un-toggle removes without collateral damage.
4. **`selectedChoicesSurviveNavigationAwayAndBack`** — navigate forward, navigate back, selection survives the engine round-trip.

Verified by reverting the fix, running the tests, watching all 4 fail, restoring, running again, watching all 4 pass. This is the minimum bar for "a test exists": can it detect the bug it's supposed to protect against?

## Recommended Policy

This isn't a one-off. Every ViewModel in `app/src/commonMain/kotlin/org/commcare/app/viewmodel/` has this gap. Based on the bug count so far, the ViewModel layer should be presumed to have similar latent bugs in any state machine that interacts with the engine. Specific call-outs:

1. **Audit the placeholder tests and mark them `@Ignore("TODO: real test pending")` or delete them.** They are actively misleading. If a new contributor runs `./gradlew :app:jvmTest` and sees 40 green tests in `viewmodel/`, they will think the layer is covered. It is not.

2. **Every new ViewModel test file must construct a real ViewModel and call real methods.** If you can't do that without plumbing, the plumbing is the test fixture — build it once, reuse it. `ViewModelIntegrationTest.kt` has the right pattern (load XML, build `FormDef`, build `FormEntrySession`, build `FormEntryViewModel`, exercise). The rest of the suite should look like it.

3. **Every ViewModel state field that is set from multiple code paths needs a "does it survive refresh?" test.** The bug class here is always the same: field A is set from path P1, but refresh path P2 rebuilds the state and forgets A. `selectedChoices` is one example; `draftTexts` in the same file already has this kind of test (#394 landed it). Other candidates: `constraintMessage` (already preserved via `priorConstraints` map), `persistentTileData`, `isRepeatPrompt`, anything else that lives in `FormEntryViewModel` and can be mutated outside of `updateQuestions()`.

4. **Expect more bugs of this shape as Phase 9 advances.** Wave 5a is the first wave to stress-test the ViewModel layer across a real form with mixed question types. Every new question type in Wave 5/6/7 will potentially surface analogous state-drop bugs. Budget for them. The original rule (`every actual fun gets a test`) was about platform impls; the analogous rule for ViewModels is `every mutable state field gets a round-trip test`.
