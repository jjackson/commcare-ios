## Key Findings

### 1. PrototypeFactory hash divergence (Wave 0)

**Problem:** iOS `PrototypeFactory` used a 4-byte XOR hash while JVM used 32 bytes (first 32 bytes of reversed class name UTF-8). Serialized data from one platform couldn't deserialize on the other.

**Fix:** Rewrote iOS hash to match JVM: reverse class name â†’ UTF-8 encode â†’ copy first 32 bytes, zero-pad if shorter. Added `PrototypeFactoryHashTest` with 8 cross-platform tests to prevent regression.

**Lesson:** Any platform-specific `actual` implementation must be validated against the `expect` contract with cross-platform tests. Unit tests on one platform alone won't catch divergence.

### 2. Serialization stubs hiding real failures (Wave 0)

**Problem:** iOS `writeTagged`/`readTagged` in `SerializationHelpers.kt` threw `UnsupportedOperationException` stubs instead of delegating to `ExtWrapTagged.writeTag()`/`ExtWrapTagged.readTag()` + `ExtUtil.write()`/`ExtUtil.read()`. This silently broke tagged serialization on iOS.

**Fix:** Replaced stubs with proper delegation. Added `KClassRegistry` for iOS class nameâ†’KClass lookup (replacing JVM's `Class.forName`).

**Lesson:** Stub implementations should fail loudly in tests, not silently in production paths. Better: write cross-platform round-trip tests that exercise the actual serialization code.

### 3. Javaâ†’Kotlin test conversion: `!!` on nullable returns (Wave 3)

**Problem:** `ExternalizableTest.doTests` NPE. The Javaâ†’Kotlin conversion introduced `val deser: Any = ... ExtUtil.read(...)!!` but `ExtWrapNullable` legitimately returns `null` when wrapping a null value. Java's `Object` type allowed null; Kotlin's `Any` with `!!` does not.

**Fix:** Changed to `val deser: Any? = ... ExtUtil.read(...)` (removed `!!`, made type nullable).

**Lesson:** When converting Java to Kotlin, `Object` return types often need `Any?` not `Any`. The `!!` operator should only be used when you've verified the value is never null â€” don't add it mechanically to satisfy the type checker.

### 4. `Class<*>` vs `KClass<*>` in wrapper constructors (Wave 3)

**Problem:** KMP wrapper classes (ExtWrapList, ExtWrapMap, etc.) have deserialization constructors taking `KClass<*>`, not `Class<*>`. JVM compat functions in `ExtWrapJvmCompat.kt` bridge this (`fun ExtWrapList(type: Class<*>) = ExtWrapList(type.kotlin)`). But when converting test files, `ExtWrapNullable(String::class.java)` matched the `constructor(val: Any?)` serialization constructor instead of the intended `constructor(type: KClass<*>?)` deserialization constructor.

**Fix:** Changed to `ExtWrapNullable(String::class)` to match the KClass constructor directly. The JVM compat functions handle `Class<*>` for Java callers; Kotlin callers should use `KClass<*>` directly.

**Lesson:** In KMP code, prefer `::class` (KClass) over `::class.java` (Class) in Kotlin callers. The JVM compat bridge functions exist for Java backward compatibility only.

### 5. Getter/setter property conversion safety (Wave 4)

**Safe to convert:**
- Private field + public getter with no logic â†’ `var field ... private set`
- Constructor parameter + getter â†’ constructor `val` property
- Builder-set fields â†’ `var field ... internal set`

**Not safe to convert:**
- `open` classes where subclasses override getters (TreeElement, Case, FormDef)
- Interface method implementations (getKey/getValues in QueryData)
- Getters with computation, null-coalescing, or parameter-based logic
- Classes with `@JvmField` on `open` properties

**Lesson:** Property conversion is safe for final classes with simple accessors. For `open` classes or interface implementations, the generated bytecode may differ from the original explicit method, breaking subclass overrides.

### 6. Parallel agent strategy for bulk conversion

**What worked:** Running 2-4 agents in parallel for independent file groups (e.g., case tests vs session tests vs form tests). Each agent converts files, then a compilation pass catches cross-cutting issues.

**What didn't work:** Having agents fix compilation errors independently without seeing each other's changes â€” they sometimes made conflicting fixes to shared files.

**Lesson:** Parallelize by file group (no shared files), then serialize the fix pass. The compile-fix-compile loop is inherently sequential.
